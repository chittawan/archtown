# วิเคราะห์การใช้งาน Auto Sync ไป Cloud

## โครงสร้างข้อมูลที่ sync อยู่ตอนนี้

- **รูปแบบ**: JSON หนึ่งไฟล์ ประกอบด้วย
  - `schema_version`, `version`, `updated_at`
  - `tables`: object ที่มี 9 ตาราง (projects, project_teams, project_topics, project_sub_topics, project_sub_topic_details, org_teams, org_team_children, capability_order, caps, cap_projects)
- **การอัปโหลด**: ส่ง **ทั้ง payload** ทุกครั้ง (full snapshot)
- **Server**: เก็บหนึ่งไฟล์ต่อ user ที่ `data/sync/<userId>/backup.json` และมี version check (409 ถ้า Cloud มี version ใหม่กว่า)

---

## ขนาดข้อมูลโดยประมาณ

| สถานะข้อมูล | ขนาดโดยประมาณ (JSON) |
|-------------|------------------------|
| ว่าง / โปรเจกต์เดียว เล็ก | ~2–5 KB |
| ใช้จริงปานกลาง (โปรเจกต์ 10–20, ทีม/หัวข้อ/งานหลายสิบ) | ~20–80 KB |
| ใหญ่ (โปรเจกต์หลายสิบ, รายการร้อย+ มีข้อความยาว) | ~100–500 KB |
| ใหญ่มาก (รายการพัน+ หรือ description ยาวมาก) | ~1–2 MB+ |

จากตัวอย่างจริงใน repo (backup ของ user หนึ่ง) ขนาดอยู่ระดับ **หลายหมื่น bytes (ราว 30–50 KB)** สำหรับข้อมูลที่ค่อนข้างเต็มแล้ว ดังนั้นโดยทั่วไป **การส่งทั้งหมดครั้งละหนึ่งไฟล์ยังไม่ถือว่า “เยอะเกินไป”** สำหรับการ sync บนเครือข่ายปกติ

---

## ส่งข้อมูลไปทั้งหมดจะเยอะเกินไปไหม?

- **โดยรวม: มักจะไม่เยอะเกินไป**  
  สำหรับการใช้งาน ArchTown ทั่วไป (โปรเจกต์/ทีม/งานในระดับสิบถึงร้อยรายการ) ขนาดอยู่ที่หลักสิบ KB ถึงหลักร้อย KB การอัปโหลด full snapshot เป็นระยะ (เช่น หลัง save, หรือ auto sync ทุก 1–2 นาที) รับได้ทั้งด้านปริมาณข้อมูลและจำนวนครั้งที่ส่ง

- **เมื่อไหร่ถึงจะเริ่ม “เยอะ”**  
  - เปิด auto sync บ่อยมาก (เช่น ทุก 5–10 วินาที) และข้อมูลใหญ่ (หลักร้อย KB ขึ้นไป) →  traffic และโหลด server สูง  
  - ข้อมูลใหญ่มาก (หลัก MB) และเครือข่ายช้า → รู้สึกอืดเวลา sync

ดังนั้น **ไม่จำเป็นต้องเปลี่ยนเป็น “ส่งแค่ส่วนที่เปลี่ยน” ก่อน** ถ้าทำ auto sync แบบ **ส่งทั้งหมด แต่ throttle/debounce ให้ส่งไม่บ่อยเกินไป** (ดูหัวข้อถัดไป)

---

## แนวทางทำ Auto Sync

### 1) Auto Sync แบบส่งทั้งหมด (Full snapshot) + Debounce (แนะนำทำก่อน)

- **ไอเดีย**: หลังมี “การเปลี่ยนแปลง” (เช่น save โปรเจกต์ / ทีม / capability) ไม่ต้องอัปโหลดทันที แต่รอให้ไม่มีกิจกรรมต่ออีก **ช่วงหนึ่ง (เช่น 30–60 วินาที)** แล้วค่อย export ทั้งหมดแล้วส่งไป Cloud ครั้งเดียว
- **ข้อดี**
  - ใช้ logic เดิมได้เลย (export ทั้งหมด → POST `/api/sync/upload`)
  - ไม่ต้องมี delta / version ระดับแถว
  - ลดจำนวน request และปริมาณรวมที่ส่งเมื่อผู้ใช้แก้ถี่ๆ
- **ข้อเสีย**
  - แต่ละครั้งที่ sync ยังส่งขนาดเท่า full snapshot (แต่ส่งน้อยครั้งลง)
- **ข้อควรระวัง**
  - ยังต้องใช้ `version` บน server เหมือนเดิม เพื่อกันกรณีหลายแท็บ/หลายอุปกรณ์ (ถ้า Cloud มี version ใหม่กว่า จะได้ 409; แล้วให้ผู้ใช้เลือก overwrite หรือไม่)

**สรุป**: ถ้าต้องการ “Auto sync to cloud” แบบไม่ซับซ้อน **ส่งข้อมูลไปทั้งหมดแบบ full snapshot แต่ทำ debounce (และถ้าอยาก ลดความถี่ด้วย interval สูงสุด เช่น ไม่เกิน 1 ครั้งต่อ 1–2 นาที)** ก็เพียงพอและไม่ “เยอะเกินไป” สำหรับการใช้งานทั่วไป

---

### 2) ลดขนาดต่อครั้ง (ถ้าอยาก optimize ภายหลัง)

- **บีบอัด (gzip)**  
  - Client ส่ง `Content-Encoding: gzip` + body เป็น gzip  
  - Server ต้องรองรับการอ่าน body ที่ถูก gzip (หรือใช้ middleware แกะ gzip ให้)  
  - โดยทั่วไปช่วยลดขนาดได้มาก (มักเหลือประมาณ 20–30% ของขนาดเดิม)

- **ส่งเฉพาะตารางที่เปลี่ยน (Delta)**  
  - ต้องออกแบบว่า “เปลี่ยน” หมายถึงอะไร (เช่น เก็บ `updated_at` ระดับตารางหรือแถว)  
  - Server ต้องมี API รูปแบบใหม่ (เช่น PATCH, หรือ endpoint “merge delta”) และ logic รวมข้อมูล  
  - ซับซ้อนกว่าแบบ full snapshot มาก จึงเหมาะทำในระยะหลังถ้าข้อมูลโตมากจริงๆ

---

### 3) ความถี่ที่สมเหตุสมผลสำหรับ Auto Sync

- **Debounce หลังการแก้ไข**: 30–60 วินาที หลัง save ครั้งล่าสุด แล้วค่อย export + upload ครั้งเดียว  
- **ขีดบนของความถี่**: เช่น ไม่เกิน 1 ครั้งต่อ 1–2 นาที แม้จะมีการแก้ต่อเนื่อง (รวมถึงกรณีหลายแท็บ)  
- **เมื่อเปิดแอป**:  
  - โหลดจาก Cloud ครั้งเดียวเมื่อเข้าแอป (เหมือน “Restore จาก Cloud” ที่มีอยู่)  
  - หรือถ้าต้องการ “always latest” อาจเช็ค version กับ server แล้วถามว่าจะดึงจาก Cloud มาแทนหรือไม่

## สรุปคำตอบ

| คำถาม | คำตอบสั้น |
|--------|------------|
| ส่งข้อมูลไปทั้งหมดจะเยอะเกินไปไหม? | **โดยทั่วไปไม่** สำหรับขนาดข้อมูล ArchTown ปกติ (หลักสิบ KB ถึงหลักร้อย KB) |
| Auto sync ควรทำแบบใด? | **เริ่มจาก “ส่งทั้งหมด (full snapshot) + debounce”** เช่น รอ 30–60 วินาที หลังการแก้ไขแล้วค่อยอัปโหลดครั้งเดียว และจำกัดความถี่ (เช่น ไม่เกิน 1 ครั้งต่อ 1–2 นาที) |
| เมื่อไหร่ถึงต้องคิด Delta / แค่ส่วนที่เปลี่ยน? | เมื่อข้อมูลโตมาก (หลัก MB) หรือต้องการลด traffic ลงมากจริงๆ ค่อยพิจารณา delta หรือการบีบอัด (gzip) เป็นขั้นถัดไป |

ถ้าต้องการให้ช่วยออกแบบหรือเขียนโค้ดส่วน “Auto sync (debounced full upload)” ในแอป (จุดที่เรียก upload, debounce timer, และการจัดการ 409) บอกได้เลยว่าจะให้ผูกกับเหตุการณ์ไหน (เช่น หลัง save โปรเจกต์/ทีม/capability อย่างเดียว หรือรวมกับ IndexedDB persist ด้วย).
